octo-server → 前端 API（会话内搜索）

遵循 octo-openapi-dev-skill 规范（R1 envelope / R2 错误码 / R3 时间字段 / R5 分页 / R6 URL / R8 字段 / R10 operationId / R13 swag）。四个独立端点按资源动作拆分，统一 cursor 分页 + envelope 包络。

0. 概览

会话内搜索按 skill R6 「一资源一端点」 原则拆为 4 个独立端点（均挂 Bearer 鉴权 + Space 中间件）：

端点operationId（R10）用途关键词
POST/v1/messages/_searchmessage.search文本 / 转发卡 / 引用消息（外层概要展示）必填
POST/v1/messages/_search_mediamessage.search.media图片 / 视频时序列表（瀑布）禁传
POST/v1/messages/_search_filesmessage.search.files按文件名匹配（无 keyword 时浏览全部）可选
POST/v1/messages/_search_allmessage.search.all聚合 message + file 单列时间倒序必填

点击命中后定位仍复用 POST /v1/message/channel/sync，前端用响应里的 message_seq 作 hitSeq 拉上下文，详见 §3。

资源段用复数 messages，动作段用 _ 前缀（R6 A.2）；与既有 POST /v1/message/search（WuKongIM 透传的全局关键词搜索，单数 legacy 命名空间）**路径不冲突**，二者并存。

1. 通用约定

1.1 鉴权（R4）

所有路由挂 AuthMiddleware + SpaceMiddleware；请求头 Authorization: Bearer <token>。当前登录 id 由 token 推导，请求体不传 login_*。Space 上下文由 X-Space-ID 头或同名 query 提供。

1.2 响应 envelope（R1）

成功 envelope.CursorList[T] → { "data":[...], "pagination":{ "has_more":bool, "next_cursor":"..." } }；失败 envelope.Error → { "error":{ "code","message","details","hint" } }。不再使用 {status,msg,data} 旧包络；后续示例只展示包络内部。

1.3 时间字段（R3）

统一 _at 后缀 + RFC3339 字符串：响应 sent_at（"2026-06-08T14:20:10Z"）；请求筛选 filters.sent_at_from / filters.sent_at_to（可只到天 "2026-06-01"，按用户本地时区扩展为含起含止）。swag 标 swaggertype:"string,date-time"。

1.4 分页（R5，cursor）

请求体顶层 cursor（首屏空串或省略）+ page_size（默认 20，上限 100）。响应 pagination.has_more = (next_cursor != "")，二者同义都返。next_cursor 为不透明字符串，客户端原样回传不可解析。

1.5 排序 / 1.6 高亮

请求体 sort 枚举："time_desc"（默认）/ "time_asc" / "relevance"；_search_media 不允许 relevance。snippet 字段由服务端注入 <mark>，单片段最长 120 字；_search_media 不返回 snippet；_search_files 高亮由前端在 file_name 上自行处理（服务端不返 marked 版本）。

1.7 错误码（R2 · 12 项固定 enum）

所有失败响应 error.code 来自 skill 规定的 12 项 enum；业务子分类走 error.details 结构化字段，不新造 code。

2.1 POST/v1/messages/_searchmessage.search

命中文本消息 / 转发卡 / 引用消息，定位指向外层消息。keyword 必填非空。

服务端仍在 ES 索引内对转发卡子消息正文 / 引用原消息正文做召回（命中率不变），但 API 层仅返回外层消息 + 单 snippet（来自外层文本或服务端从内层匹配中合成的概要片段）。点击定位仍用外层 message_seq，与 _search_files / _search_media 结构对称。

请求示例

{ "channel_type":2, "channel_id":"G_abc123", "keyword":"上线方案",
  "filters":{ "sender_ids":["u_alice"], "sent_at_from":"2026-06-01", "sent_at_to":"2026-06-08" },
  "sort":"time_desc", "page_size":20, "cursor":"" }

data[] 元素字段（MessageSearchHit）

字段类型说明
message_idstring外层消息 id，定位跳转用此 id
message_seqint64必填。外层消息在所属 channel 内的单调序列号，前端用其调 sync 拉命中 + 上下文（详见 §3）
message_kindstring enum"text" / "forward"
snippetstring含 <mark> 的概要片段，来自外层文本或服务端从内层匹配中合成的片段，单片段最长 120 字
sender_id / sender_name / sender_avatar_urlstring外层消息发送人 id / 显示名 / 头像 URL（R8 _url 后缀）
sent_atstring (date-time)外层消息发送时间，RFC3339

3. 点击定位（复用 sync）

用户点击任一命中（无论端点），前端统一调 POST /v1/message/channel/sync，用搜索响应里的 message_seq（聚合 _search_all 时取 message.message_seq / file.message_seq）作为 hitSeq，按 [hitSeq - N, hitSeq + N] 一次拉命中消息 + 上下文。

sync 端点本身是 octo-server 既有 legacy 路由（路径单数 + 无 _ 前缀），不在本次重设计范围。前端按现有契约继续调用即可。

三种 channel 的参数映射

来源会话类型channel_idchannel_type
私聊（channel_type=1）对端用户 id1
群（channel_type=2）群 group_no2
子区 Thread（channel_type=5）"{group_no}____{thread_short_id}"（4 个下划线）5

请求示例

POST /v1/message/channel/sync
{ "channel_id":"<按上表>", "channel_type":<按上表>,
  "login_uid":"<当前登录 uid>", "device_uuid":"<设备 uuid>",
  "start_message_seq":<hitSeq - N>, "end_message_seq":<hitSeq + N>,
  "limit":<2*N + 1>, "pull_mode":1 }

N 推荐值：N=20（一次最多 41 条），具体由前端按上下文 UI 自决。响应按 message_seq 升序排版，把 hitSeq 那条高亮并滚动到视区中央。

注意事项

  • sync 已挂 AuthMiddleware + SpaceMiddleware，前端不附加额外鉴权。
  • 非群内用户调 sync 返回空 messages 数组（非 403），按 0 条结果处理。
  • hitSeq 可能已撤回 / 删除而在响应窗口中缺失；前端按 message_seq 排版即可，仅少高亮一条。
