# 消息搜索功能 — 已知能力边界（产品视角）

> 日期：2026-06-12（修订 v2）
> 适用版本：搜索后端 v1（feat/messages-search 已开发完成）+ 索引 mapping v1.8
> 受众：产品经理、业务方、前端 Owner
> 用途：上线前对齐"搜索能搜到什么、搜不到什么、搜出来长什么样"，并对**4 个待拍板项**给出选择

---

## TL;DR — 三句话总览

1. **能搜**：群聊 / 私聊里的文字、图片说明、文件名、合并转发卡里的子消息文本
2. **搜不到**：被引用消息的内容
3. **不完美**：撤回 / 编辑过的消息可能仍出现、图片没有专门的缩略图（用原图代替）、视频和文件一视同仁（无封面 / 无时长）、文件不能预览

---

## 一、产品体验上的关键约束（按用户感知度排序）

### ① 撤回 / 编辑过的消息可能仍被搜到 ⚠️ 上线前必决

**用户感知**：A 撤回了一条消息，B 一周内搜关键词可能仍能看到那条消息出现在搜索结果里（点进去会显示"消息已撤回"，但搜索摘要里仍可能露出原文片段）。编辑后的消息搜索时显示的是**编辑前**的内容。

**为什么**：消息撤回 / 编辑事件目前不会通知搜索索引同步更新。新消息覆盖才会刷新。

**影响窗口**：约 30 天（消息进入冷存储后索引彻底冻结）。

**2026-06-13 更新（P0 已修）**：搜索路径现已和"普通读消息"一样应用消息可见性 4 维过滤（`modules/messages_search/visibility.go::filterVisible`）：
1. `message_extra.revoke=1` → 隐藏
2. `message_extra.is_deleted=1`（管理员 / 双向删除）→ 隐藏
3. `message_user_extra.message_is_deleted=1`（当前用户删的）→ 隐藏
4. `channel_offset.message_seq >= 命中 seq`（用户清空过会话）→ 隐藏

也就是说，**用户自己删除 / 清空 / 管理员删 / 撤回**这 4 种情况下命中已不会再返回——和 IM 客户端的"会话页"展示完全一致。本条剩下的差异只剩"撤回事件没及时刷到 OS 索引导致 OS 命中预过滤偏少（命中后仍被 MySQL 4 维过滤兜底）"，**不会向用户泄漏内容**，只是少量浪费一次过滤计算。

**产品决策**：~~A. 本期接受~~ → **A 已落地**，"已撤回 / 已编辑前的内容"提示文案仍建议保留覆盖编辑后旧值场景。

---

### ② 图片的 thumb_url 实际是原图 URL

**用户感知**：在"图片/视频"专搜结果页（瀑布流），图片项响应中 `thumb_url` 字段返的是**原图 URL**，不是专门生成的小尺寸缩略图。

**为什么**：业务消息里只存原图 URL，没有专门的缩略图字段。为了让前端拿到的 `thumb_url` 总是可用（避免空字段需要特判），服务端直接返原图 URL。

**影响**：加载流量比预期大，瀑布流首屏渲染速度受 CDN 性能影响。

**前端建议在 thumb_url 上加 CDN 尺寸参数**（如 `?imageView2/2/w/320`）做客户端缩略。

**长期解锁**：客户端发送规范统一要求填专门的缩略图 URL，indexer mapping 加对应字段。**不在本期范围**。

---

### ③ 视频实际作为文件发送，没有封面 / 时长 / 尺寸信息

**用户感知**：用户发的视频在 IM 里显示成"文件"附件（带视频图标），搜索时也按文件搜索，结果里**没有封面预览图、没有视频时长、没有视频尺寸**。视频消息和其他文件类型（文档、压缩包等）展示一致。

**为什么**：当前业务消息发送链路里，视频走的是文件上传路径（ContentType=8 文件），而不是独立的视频消息类型（ContentType=5）。索引 mapping 虽然预留了视频字段（cover / second / width / height），但**业务 payload 实际上不会填这些字段**。

**影响**：
- 用户搜不到"我上次发的那个 5 分钟会议录像"——因为没有时长信息可以筛
- 视频文件在搜索结果里显示成普通文件（文件名 + 大小 + 扩展名），无法快速识别"哦这是个视频"
- "图片/视频"专搜瀑布流里，视频项实际**搜不到**（DSL 限定 ContentType=2/5，视频走 8 走的是文件搜索）

**前端建议**：在"图片/视频"专搜入口下加一行说明"视频暂以文件形式搜索，请在文件搜索中查找"。或者把"图片/视频"改名"图片"。

**长期解锁**：业务 / 客户端补独立视频消息类型 + 发送时带封面 / 时长 / 尺寸字段。**不在本期范围**。

---

### ④ 文件不能预览（preview_url 永远为空）

**用户感知**：在"文件"专搜结果列表里，每个文件项前端按"不支持预览"分支展示，点击只能下载。

**为什么**：业务消息 payload 里没有预览 URL 字段，octo 也没接入文档预览服务（如 office365 / wps cloud preview）。

**影响**：和 IM 内置文件查看体验一致（IM 里点文件本来就是下载），用户预期问题不大。

**长期解锁**：业务接入预览服务 + 客户端发送时把预览链接写进消息。**不在本期范围**。

---

### ⑤ 引用消息的被引内容搜不到

**场景**：A 引用了 B 一周前发的"明天 10 点开会"，并回复"收到"。用户搜"开会"——
- ✅ 能搜到 B 的原消息
- ❌ 搜不到 A 的引用回复（虽然引用块里包含"开会"两字）

**为什么**：v4.2 PRD 已决策"引用消息不单独索引"——理由是搜原消息已经能定位到上下文。

**前端表现**：用户搜"开会"看到 B 的原消息，点进去能看到 A 的引用回复在下方，**信息不丢**，只是搜索路径不同。

**长期解锁**：mapping 加 reply 字段索引，需 indexer 改造。**v4.2 已显式决策不做**。

---

### ⑥ 不能跨群搜索 / 不能"搜索我的所有消息"

**用户感知**：搜索必须先指定一个群或一个 DM 会话，**没有"搜索全平台"或"搜我所有群"的入口**。

**为什么**：本期 spec 明确了"单会话搜索"范围。

**长期解锁**：跨群搜索需要新一版 API + 鉴权 + 性能验证（>1000 个群的用户性能边界）。**不在本期范围**。

---

### ⑥' Person/DM 搜索按 Space 隔离（P0 修复）

**用户感知**：私聊搜索只返回当前 Space 内 A↔B 的消息；切换到另一个 Space 后，同一对 A↔B 的私聊在那个 Space 的消息是另一组结果，不会跨 Space 互相露出。

**为什么**：私聊在 IM 里用 `fakeChannelID(uid_a, uid_b)` 标识，channel_id **本身不带 space 维度**，跨 Space 复用同一对会话；DM 的 Space 归属由 message payload 里的 `space_id` 字段定义。本期已在 OS DSL 里对 p2p 加 `spaceId` term filter，请求必须经由 `SpaceMiddleware`（`X-Space-ID` header 或 `space_id` query），缺失即视为不可见，返回 NOT_FOUND（resource=channel）。

**依赖**：OS 文档需带 `spaceId` keyword 字段。在 wukongim-message-indexer 完成"读取 `payload.space_id` → 写入 OS doc.spaceId"改造**之前**，老索引文档无该字段，term filter 命中为 0 —— 这是有意设计的 fail-closed 行为，避免"老索引被默认放过"。indexer 升级 + 存量 backfill 之后命中即恢复。

**Group / Thread 不受影响**：群聊 / 子区 channel_id 已隐含 Space（不同 Space 的群 channel_id 不同），不再叠加 spaceId filter。

**配置**（rollout 期专用）：

| 环境变量 | 默认 | 含义 |
|---|---|---|
| `OCTO_SEARCH_REQUIRE_SPACE_ID` | `true` | p2p 搜索必须带 spaceID。`false` 仅在 indexer 没有写 `spaceId` 之前的过渡窗口使用，每次 p2p 请求都会 WARN 日志报警 |

**长期**：indexer/mapping v1.9 落地 + 存量回填后，永久维持 `OCTO_SEARCH_REQUIRE_SPACE_ID=true`。

---

### ⑥'' 频道访问门禁与读路径对齐（P0 修复 / 2026-06-13）

**用户感知**：从前面 4 个 `/v1/messages/_search*` 端点，能搜到的范围严格等于"在 IM 客户端能进会话页看到消息"的范围；过去能绕过的几种场景已全部封死：

| 维度 | 旧行为（已修复前） | 新行为（已修复） |
|---|---|---|
| **群已解散但成员行残留** | 偶发场景下仍可搜到群历史 | 一律 NOT_FOUND（在 `ExistMemberActive` 之前先看 `group.status==Disband`） |
| **子区已删除（status=3）** | 解析 channel_id 后只查父群成员，能搜到已删子区 | `thread.GetThread` 先校验子区状态，缺失/删除 → NOT_FOUND |
| **私聊好友关系已断 / 双向拉黑** | 凡同 Space 用户都能输入对方 uid 搜出私聊历史 | 走和 `/messages/channel-files` 同样的 `IsFriend` + 双向 `ExistBlacklist` 门禁，不满足 → NOT_FOUND |

**为什么**：搜索路径过去只检查"群成员"门禁；其他三个维度由 IM 客户端的"会话列表 / 会话页"间接拦下。从 API 层看，搜索是绕过这层防线的"读路径替身"，必须自己重做这些检查。

**实现**：`modules/messages_search/authz.go` 拆分为 `checkP2PAccess` / `checkGroupAccess` / `checkThreadAccess`，各自走和读路径相同的 fail-closed 模板。任意 DB 查询出错均直接 INTERNAL_ERROR（不会"放过去"），单一异常：thread `GetThread` 错误三合一映射为 NOT_FOUND（避免"是 DB 挂了 vs 子区不存在"的枚举 oracle）。

## 二、人名 / 头像相关约定

### ⑦ 搜索结果显示的发送人姓名是当前姓名，不是发送时的姓名

**场景**：A 一年前以"张三"的名字发过一条消息，今天 A 改名"张三丰"。用户搜到这条消息，发送人显示的是"张三丰"。

**为什么**：取的是 user 表当前值，不存历史快照。

**影响**：极少数情况下用户会困惑"我记得当时是张三发的"。

**产品建议**：在 PRD 里加一句约定"搜索结果显示发送人当前姓名，可能与历史显示不同"，无需技术改动。

---

### ⑧ 群里搜索显示的姓名优先用群备注

**约定**：群里搜结果显示 `member.remark > user.name`（如果当前用户给该成员设了群备注，显示备注；否则显示用户名）。这与群聊里看到的名字一致，**符合用户预期，无需特别说明**。

## 五、不需要决策但建议在 PRD / 帮助中心里说明的约定

1. "搜索支持文字、图片说明、文件名、合并转发卡内文本。"
2. "视频文件以文件形式参与搜索，不显示视频封面 / 时长。"
3. "搜索结果可能包含 30 天内已撤回 / 已编辑前的消息内容。"
4. "搜索结果按时间倒序排列；搜索关键词模式下也优先按时间，再按相关度。"
5. "图片缩略图基于原图加载，可能比预期慢。"
6. "文件搜索结果暂不支持在线预览，点击直接下载。"
7. "搜索范围限定在当前会话（群聊 / 私聊），不支持跨会话搜索。"
8. "搜索结果显示的发送人姓名为该用户当前姓名 / 群备注。"
