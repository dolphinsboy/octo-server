# WuKongIM 消息搜索 — OpenSearch Mapping (v1.8)

> 日期：2026-06-12
> 范围：`wukongim-message-indexer` v1.8（在 v1.7 mergeForward C+ 基础上加媒体元字段）。
> 权威源：仓库 `internal/indexer/mapping.go` 的 `IndexTemplateJSON` 常量（本文档由该常量直出，非手抄）。
> 适用：OpenSearch 2.18+，装 `analysis-ik` 分词器（版本与集群一致）。

---

## 1. v1.8 相对 v1.7 的字段变更

只在 `payload.<X>` 子对象里加字段，外层结构不动：

| 子对象 | 新增字段 | 类型 | 来源（业务原生 payload 字段） |
|---|---|---|---|
| image | width / height | integer | ImageContent.decodeJSON: width/height |
| video | width / height / second | integer | VideoContent.decodeJSON: width/height/second（秒长，**不是 ms**） |
| file | size / extension | long / keyword | FileContent.decodeJSON: size(byte)/extension |
| mergeForward | childCount | integer | indexer 算 = len(投影后 msgs) |

说明：
- 字段贴业务原名原单位（second 秒 / size byte / extension 原样不转小写）。
- 不造业务没有的字段：**无 thumbUrl / durationMs / previewUrl**——A 文档要的展示字段（thumb_url / duration_ms / file_size_bytes / file_ext）由 octo-server BFF 层从这些原字段映射/换算。
- 新增字段都是 integer/long/keyword，不参与 ik 分词，不进 mergeForward.msgs.searchText。

## 1.1 v1.9 待加（indexer 侧改造，跨仓库依赖）

> 状态：octo-server `feat/messages-search` 已上线 p2p Space scope 过滤（P0 修复，PR #361）；indexer 侧改动尚未排期。

**背景**：私聊 (`channel_type=1`) 在 IM 里用 `fakeChannelID(uid_a, uid_b)` 标识 channel_id，**channel_id 本身不带 space 维度**，跨 Space 复用同一对会话。OS DSL 层已经按 `channelId + spaceId` 双维度过滤；但当前 v1.8 mapping **没有 `spaceId` 字段**，indexer 也没把 `payload.space_id` 写入 OS doc。在 indexer 没补这个字段之前，p2p 搜索的 spaceId term filter **匹配不到任何文档** —— 这是 octo-server 端有意的 fail-closed 行为，避免老索引被默认放过；但同时也意味着 p2p 搜索结果会暂时为空。

**所需 indexer / mapping 改动**：

| 字段 | 类型 | 来源 | 备注 |
|---|---|---|---|
| `spaceId` | keyword | `payload.space_id` | 仅 channel_type=1 (p2p) 写入；group/thread 可省略，channel_id 已隐含 space |

**rollout 顺序**：
1. indexer 升级，新写入文档带 `spaceId`；
2. 对存量索引文档做 backfill（`UpdateByQuery` 或重建）；
3. 全部完成后，octo-server 维持 `OCTO_SEARCH_REQUIRE_SPACE_ID=true`（默认值）。

**rollout 过渡期临时方案**：在 indexer / backfill 完成前，octo-server 端可临时把 `OCTO_SEARCH_REQUIRE_SPACE_ID=false`（每次 p2p 请求 WARN 一条日志），这会绕过 spaceId filter 让老索引可用 —— 但代价是 P0 跨 Space 隔离风险重新打开，**仅供 SRE 在切换窗口期使用**。

## 4. 升级落地（已在测试环境执行 2026-06-12）

alias 双写零停机：

1. PUT 上面的 v1.8 template
2. 新建引导索引 `wukongim-messages-...-000002`
3. write 别名原子切到 000002（`is_write_index:true`）
4. `_reindex` 000001→000002，painless 从 payloadRaw 回填新字段（媒体字段是直接复制，painless 一把梭；mergeForward.childCount = msgs.length）
5. read 别名收敛到 000002，老索引保留 7 天回滚
6. indexer 滚动重启用新镜像

回滚：read 别名切回老索引，30s 见效。

### 4.1 reindex 回填 painless（参考）

```painless
def raw = ctx._source.payloadRaw;
if (raw == null || ctx._source.payload == null) { return; }
def p = ctx._source.payload;
int t = p.type == null ? 0 : (int)p.type;
if (t == 2 && p.image != null) {
  if (raw.width != null)  p.image.width  = (int)raw.width;
  if (raw.height != null) p.image.height = (int)raw.height;
} else if (t == 5 && p.video != null) {
  if (raw.width != null)  p.video.width  = (int)raw.width;
  if (raw.height != null) p.video.height = (int)raw.height;
  if (raw.second != null) p.video.second = (int)raw.second;
} else if (t == 8 && p.file != null) {
  if (raw.size != null)      p.file.size      = (long)raw.size;
  if (raw.extension != null) p.file.extension = (String)raw.extension;
} else if (t == 11 && p.mergeForward != null && p.mergeForward.msgs != null) {
  p.mergeForward.childCount = p.mergeForward.msgs.length;
}
```

## 5. v1.10 待加（pending — 跟踪 follow-up issue）

> 这一节列在搜索 P0 visibility fix（2026-06-13）期间发现的 mapping / indexer 漏洞。修复都涉及 indexer 而不在 search 范围内，因此与本期搜索后端解耦——临时由 `modules/messages_search/visibility.go::filterVisible` 在 server 端 post-filter 兜底。

### 5.1 `revoked` 字段在撤回事件后未刷新

**症状**：`payload.type=99 (CMD)` 的撤回 / 删除事件不会触发对 OS 中已存在 doc 的 partial update，导致：
- `MustNot Term(revoked=true)` 这一 best-effort filter 命中率 < 5%（看监控）
- 真正的隔离靠 server 端 `filterVisible` 走 MySQL `message_extra.revoke=1` 兜底

**v1.10 计划**：indexer 收到撤回事件时，对原 doc 执行 `_update { "doc": { "revoked": true } }`（routing 已知）。落地后 OS 端命中可见率 ≈ 100%，server 端 post-filter 退化为冗余防线（保留以防漂移）。

### 5.2 `is_deleted` 字段缺失

**症状**：管理员删 / 双向删除（`message_extra.is_deleted=1`）目前**完全不进 OS**——OS doc 没有对应字段。server 端只能按 `filterVisible` 在 MySQL 侧过滤。

**v1.10 计划**：mapping 加 `"isDeleted": { "type": "boolean" }`；indexer 对应事件触发 partial update。落地后 search DSL 可加 `MustNot Term(isDeleted=true)` filter，OS 命中先行过滤。

### 5.3 `messageUserExtra` / `channelOffset` 不进 OS

**症状**：用户级删除（`message_user_extra.message_is_deleted=1`）和清空会话（`channel_offset.message_seq`）是**每用户**信号，不适合写到（共享的）OS doc 里。

**永久结论**：这两路信号继续靠 server 端 post-filter 处理（每个搜索请求一次 MySQL 批量 IN 查询），不进 mapping。
